    **Create src/Wcmux.App/Notifications/NotificationService.cs:**

    1. Create the Notifications directory under Wcmux.App
    2. Implement NotificationService class:
       - Constructor takes `IntPtr hwnd` (for FlashWindowEx)
       - `void ShowAttentionToast(string tabId, string tabLabel, string paneId, string paneTitle)`:
         - Build notification with `AppNotificationBuilder`
         - Add arguments: action=focusPane, tabId, paneId
         - Add text lines: "wcmux -- Tab: {tabLabel}" and paneTitle
         - Set `notification.Tag = paneId` and `notification.Group = tabId` for targeted removal
         - Call `AppNotificationManager.Default.Show(notification)`
       - `void FlashTaskbar()`:
         - P/Invoke `FlashWindowEx` with `FLASHW_ALL | FLASHW_TIMERNOFG` flags
         - Use stored hwnd
       - `async Task DismissAllAsync()`:
         - Call `await AppNotificationManager.Default.RemoveAllAsync()`
       - Include P/Invoke declarations for FlashWindowEx:
         - `[DllImport("user32.dll")] static extern bool FlashWindowEx(ref FLASHWINFO pwfi)`
         - FLASHWINFO struct with cbSize, hwnd, dwFlags, uCount, dwTimeout
         - Constants: FLASHW_ALL = 3, FLASHW_TIMERNOFG = 12

    **Modify src/Wcmux.App/App.xaml.cs:**

    1. Add a static `Action<string, string>? OnNotificationActivated` callback (tabId, paneId)
    2. In OnLaunched (before creating MainWindow):
       - Subscribe `AppNotificationManager.Default.NotificationInvoked += OnNotificationInvoked`
       - Call `AppNotificationManager.Default.Register()`
       - CRITICAL: Subscribe BEFORE Register per research pitfall #1
    3. Add `OnNotificationInvoked` handler:
       - Extract tabId and paneId from `args.Arguments`
       - Call `OnNotificationActivated?.Invoke(tabId, paneId)`
    4. Add shutdown cleanup:
       - Override `OnExit` or use Window.Closed: call `AppNotificationManager.Default.Unregister()`

    **Modify src/Wcmux.App/MainWindow.xaml.cs:**

    1. Add `private bool _isWindowFocused = true` field
    2. Add `private NotificationService? _notificationService` field
    3. Repurpose the existing `Activated` event handler (currently used only for one-time init):
       - After the existing init logic, keep the handler subscribed (do NOT unsubscribe with `Activated -= OnActivated`)
       - Track focus: `_isWindowFocused = args.WindowActivationState != WindowActivationState.Deactivated`
       - On regain focus (transition to focused): call `_notificationService.DismissAllAsync()`
       - NOTE: The current code unsubscribes `Activated -= OnActivated` after first call. Change this:
         move the one-time init into a separate method called from the handler on first invocation, but keep the handler subscribed for focus tracking
    4. Create NotificationService after window init:
       - Get HWND via `WinRT.Interop.WindowNative.GetWindowHandle(this)`
       - `_notificationService = new NotificationService(hwnd)`
    5. Subscribe to `AttentionStore.AttentionChanged`:
       - When attention is raised (HasAttention returns true) AND `!_isWindowFocused`:
         - Look up the pane's tab and pane title for toast content
         - Call `_notificationService.ShowAttentionToast(tabId, tabLabel, paneId, paneTitle)`
         - Call `_notificationService.FlashTaskbar()`
       - When window IS focused: do not fire toast (in-app indicators from Plan 01 suffice)
    6. Subscribe to `App.OnNotificationActivated`:
       - Marshal to UI thread via DispatcherQueue
       - Activate the window (bring to foreground)
       - Call TabViewModel to switch to tabId
       - Call appropriate WorkspaceView.FocusPaneAsync(paneId) to focus the pane
       - This will trigger attention clearance via the existing focus -> ClearAttention wiring from Plan 01

    **Threading considerations:**
    - AttentionChanged fires from background thread -- marshal UI and notification calls to UI thread
    - NotificationInvoked fires from background thread -- marshal to UI thread before tab/pane operations
    - FlashWindowEx is safe to call from any thread
    - Track _isWindowFocused transitions carefully to avoid duplicate toast dismissals (pitfall #4 from research)
