<task type="auto">
  <name>Task 1: Create TabViewModel and refactor MainWindow for multi-tab</name>
  <files>
    src/Wcmux.App/ViewModels/TabViewModel.cs,

  <action>
    Create TabViewModel in Wcmux.App.ViewModels as a sealed class implementing IAsyncDisposable. It wraps TabStore + SessionManager and manages per-tab WorkspaceViewModel instances.

    TabViewModel responsibilities:
    - Owns a TabStore instance and a SessionManager reference
    - Maintains Dictionary of tabId to WorkspaceViewModel
    - CreateNewTabAsync(): Creates a session in user's home directory (Environment.GetFolderPath(UserProfile)), creates WorkspaceViewModel, registers in TabStore with FormatTabLabel(homeDir) as label. Subscribes to WorkspaceViewModel.LastPaneClosed to auto-close the tab.
    - SwitchTab(string tabId): Delegates to TabStore.SwitchTab
    - CloseTabAsync(string tabId): Calls WorkspaceView.DetachAsync on the active workspace view (if applicable), then disposes the WorkspaceViewModel, then calls TabStore.CloseTab. If LastTabClosed fires, surfaces it.
    - RenameTab(string tabId, string newLabel): Delegates to TabStore.RenameTab
    - ActiveWorkspace property: Returns the WorkspaceViewModel for the current active tab
    - Expose TabStore for the tab bar to read
    - LastTabClosed event: Surfaces TabStore.LastTabClosed for MainWindow to listen to
    - DisposeAsync: Disposes all WorkspaceViewModels

    Important: When LastPaneClosed fires on a WorkspaceViewModel, TabViewModel should call CloseTabAsync for that tab. This replaces the old direct-close-app behavior.

    Refactor MainWindow.xaml:
    - Change from single WorkspaceView to a Grid with two rows: Row 0 for TabBarView (Auto height), Row 1 for tab content area (Star).
    - The tab content area is a Grid named "TabContentArea" that will hold WorkspaceView instances (one per tab, visibility toggled).
    - Add xmlns for views namespace if not present.

    Refactor MainWindow.xaml.cs:
    - Replace single WorkspaceViewModel with TabViewModel
    - CreateInitialSessionAsync becomes CreateInitialTabAsync: Creates TabViewModel, calls CreateNewTabAsync, creates a WorkspaceView for the tab, attaches it.
    - Subscribe to TabViewModel.LastTabClosed to call Close().
    - Subscribe to TabStore.ActiveTabChanged to handle tab switching: hide current WorkspaceView (Visibility.Collapsed), show target WorkspaceView (Visibility.Visible), call UpdateContainerSize on the newly visible workspace, focus the active pane's terminal.
    - Each tab gets its own WorkspaceView instance stored in a Dictionary of tabId to WorkspaceView.
    - PaneCommandBindings.Attach needs to be called with the ACTIVE tab's WorkspaceViewModel. On tab switch, detach old bindings and attach new ones. Modify PaneCommandBindings to add a Detach method that clears keyboard accelerators, or clear and re-attach on switch.
    - OnClosed: Detach all WorkspaceViews, dispose TabViewModel, dispose SessionManager.

    Add a Detach method to PaneCommandBindings: `public static void Detach(UIElement target)` that clears all keyboard accelerators from the target. This is needed because pane commands must route to the active tab's WorkspaceViewModel.
  </action>

  <action>
    Create TabBarView as a UserControl in Wcmux.App.Views.

    TabBarView.xaml:
    - Grid with Height="32", Background="#252526" (dark terminal aesthetic per user constraint: "lightweight, not browser-heavy").
    - Inside: a ScrollViewer (HorizontalScrollBarVisibility="Hidden", VerticalScrollBarVisibility="Disabled", HorizontalScrollMode="Enabled") containing a horizontal StackPanel named "TabStrip".
    - A "+" Button (AddTabButton) at the right side: Width=32, Height=32, Background="Transparent", Foreground="#999", Content="+", HorizontalAlignment="Right".

    TabBarView.xaml.cs:
    - Attach(TabViewModel viewModel) method: Stores reference, subscribes to TabStore.TabsChanged and TabStore.ActiveTabChanged, calls RenderTabs().
    - RenderTabs(): Clears TabStrip, iterates TabStore.TabOrder, for each tab creates a tab item:
      - A Grid containing: a TextBlock for the label and a close Button ("X").
      - Active tab gets distinct background (#3c3c3c), inactive gets transparent.
      - Close button: always visible (per user constraint), click calls TabViewModel.CloseTabAsync(tabId).
      - Tab click (on the Grid, not the close button): calls TabViewModel.SwitchTab(tabId).
      - Double-click on TextBlock: Replace TextBlock with a TextBox for inline rename. On Enter or LostFocus, call TabViewModel.RenameTab(tabId, newText) and re-render. On Escape, cancel and re-render.
    - Tab item styling: FontSize=12, Foreground="#ccc", Padding(12,0,4,0) for label, Height=32, MinWidth=80.
    - Close button styling: Content="X" (plain text, not symbol), FontSize=10, Width=20, Height=20, Background="Transparent", Foreground="#888", VerticalAlignment="Center".
    - AddTabButton click: calls TabViewModel.CreateNewTabAsync(), then MainWindow needs to handle the new tab's WorkspaceView creation. Expose an event NewTabRequested or have MainWindow subscribe to TabStore.TabsChanged.

    Wire MainWindow to create WorkspaceView instances when new tabs are created:
    - In MainWindow, after TabStore.TabsChanged fires (or after CreateNewTabAsync returns), check if any tab lacks a WorkspaceView and create one.
    - Better approach: TabViewModel.CreateNewTabAsync returns the tabId. MainWindow calls a method to create and attach the WorkspaceView for that tab.

    Create TabCommandBindings as a static class in Wcmux.App.Commands:
    - Attach(UIElement target, TabViewModel viewModel, Func<Task> createTabHandler):
      - Ctrl+Shift+T: New tab (calls createTabHandler which creates tab + WorkspaceView)
      - Ctrl+Tab: Next tab (TabStore.TabOrder, find current index, wrap to next)
      - Ctrl+Shift+Tab: Previous tab (reverse of above)
      - Ctrl+1 through Ctrl+9: Switch to tab by index (1-based, Ctrl+9 always goes to last tab)
    - These are persistent (not re-attached on tab switch) because they operate on TabViewModel, not WorkspaceViewModel.

    Modify WorkspaceView.CreatePaneViewAsync to add a pane title overlay showing the current working directory.

    In the existing pane container structure (Grid > Border > TerminalPaneView), add a TextBlock overlay:
    - Create a TextBlock positioned at the top of the border area (inside the Grid that wraps the Border):
      - Text: PathHelper.TruncateCwdFromLeft(session.LastKnownCwd ?? "")
      - FontSize: 11
      - Foreground: semi-transparent light gray (Color.FromArgb(180, 200, 200, 200))
      - Margin: Thickness(8, 0, 0, 0)
      - VerticalAlignment: Top
      - HorizontalAlignment: Left
      - IsHitTestVisible: false (so clicks pass through to the terminal)
      - Tag the TextBlock with the paneId for later lookup.

    Subscribe to cwd changes to update the title dynamically:
    - After creating the TerminalSurfaceBridge (inside TerminalPaneView.AttachAsync), the bridge fires CwdChanged events.
    - WorkspaceView needs access to cwd changes. Two approaches:
      (a) TerminalPaneView exposes a CwdChanged event that WorkspaceView subscribes to.
      (b) WorkspaceView subscribes to SessionManager.SessionEventReceived for SessionCwdChangedEvent.
    - Use approach (b) since WorkspaceView already has access to SessionManager through WorkspaceViewModel.SessionManager. Filter for SessionCwdChangedEvent matching the pane's session ID, then update the TextBlock text.
    - On cwd change: look up the TextBlock by paneId, update Text to PathHelper.TruncateCwdFromLeft(newCwd).

    Important: Long paths are truncated from the left per user constraint (e.g., ".../deep/path"). The PathHelper from plan 01 handles this.

    Important: Pane cwd does NOT propagate to the tab label (per user constraint). Tab labels are static. Only pane border titles update.

    Clean up cwd subscriptions when pane views are removed (in the reconciliation loop that calls DetachAsync).
