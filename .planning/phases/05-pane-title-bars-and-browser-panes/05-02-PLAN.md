  <action>
1. Create `BrowserPaneView.xaml` UserControl with two-row Grid layout:
   - Row 0 (28px): Address bar containing:
     - Back button: FontIcon `\uE72B` (ChevronLeft), enabled bound to `CoreWebView2.CanGoBack`
     - Forward button: FontIcon `\uE72A` (ChevronRight), enabled bound to `CoreWebView2.CanGoForward`
     - Reload button: FontIcon `\uE72C` (Refresh)
     - URL TextBox: stretches to fill remaining space, dark background (#2D2D2D), light text
     - Go button: FontIcon `\uE7AD` (Forward) or text "Go"
   - Row 1 (*): WebView2 control for browser content
   - Use dark theme styling consistent with the app (#1E1E1E backgrounds, #CCCCCC text)

2. Create `BrowserPaneView.xaml.cs` code-behind:
   - `public string? PaneId { get; set; }` property
   - `InitializeWebViewAsync()`: call `WebViewEnvironmentCache.GetOrCreateAsync()` to get shared environment, then `webView.EnsureCoreWebView2Async(environment)` to initialize
   - Wire `CoreWebView2.SourceChanged` to update address bar text
   - Wire `CoreWebView2.NavigationCompleted` to update back/forward button enabled states
   - Back/Forward/Reload button click handlers call `CoreWebView2.GoBack()`, `GoForward()`, `Reload()`
   - Go button and TextBox Enter key: normalize URL (prepend `https://` if no scheme via `Uri.TryCreate`), then `CoreWebView2.Navigate(url)`
   - Handle `AcceleratorKeyPressed` event on CoreWebView2Controller to intercept app-level shortcuts (Ctrl+Shift+H, Ctrl+Shift+V, etc.) before WebView2 swallows them. When an app shortcut is detected, mark handled=false and raise a `CommandReceived` event (same pattern as terminal panes)
   - `public event Action<string>? CommandReceived` for routing keyboard shortcuts
   - Open to `about:blank` initially with address bar focused
   - `DetachAsync()` method for cleanup

3. In `WorkspaceViewModel`, add `SplitActivePaneAsBrowserAsync(SplitAxis axis)`:
   - Generate a new paneId and a sentinel sessionId prefixed with `browser:` (e.g., `"browser:" + Guid.NewGuid().ToString("N")`)
   - Do NOT create a ConPTY session -- browser panes have no terminal session
   - Do NOT add to `_paneSessions` dictionary
   - Call `_layoutStore.SplitActivePane(axis, newPaneId, sentinelSessionId)` but need to pass PaneKind
   - This requires modifying `LayoutStore.SplitActivePane` to accept an optional `PaneKind kind = PaneKind.Terminal` parameter, and propagate it to the new LeafNode

4. In `LayoutStore.SplitActivePane`, add optional `PaneKind kind = PaneKind.Terminal` parameter. Pass it through to `LayoutReducer.SplitPane` (also add parameter there). The new LeafNode should be created with `Kind = kind`.

5. In `WorkspaceView.CreatePaneViewAsync`, make PaneKind-aware:
   - Look up the LeafNode for the paneId to check its `Kind` property
   - If `PaneKind.Terminal`: create TerminalPaneView as before (get session, attach)
   - If `PaneKind.Browser`: create BrowserPaneView, set PaneId, call InitializeWebViewAsync, wire CommandReceived
   - Store browser views in a new `Dictionary<string, BrowserPaneView> _browserPaneViews` for cleanup
   - Skip session lookup for browser panes (session is null, that's expected)
   - The process name timer should skip browser panes (check Kind or check if paneId starts with "browser:" sentinel)

6. Add a browser button to the pane title bar (in `CreatePaneTitleBar` from Plan 01):
   - FontIcon `\uE774` (Globe), 20x20, positioned between split-vertical and close buttons
   - Click handler: `_viewModel.SetActivePane(paneId)` then `_viewModel.SplitActivePaneAsBrowserAsync(SplitAxis.Vertical)`
   - The browser button splits the current pane and opens a browser in the new split

7. In `WorkspaceView.DetachAsync`, also clean up `_browserPaneViews` (call DetachAsync on each).

8. In `WorkspaceViewModel.ClosePaneAsync`, handle browser panes: if `_paneSessions` doesn't contain the paneId (browser pane has no session), skip the session teardown but still call `_layoutStore.ClosePane(paneId)`.
